# Project Structure Guide

This guide defines where files should live in BiliSpider. It is intended for future AI assistants and contributors so the repository stays predictable.

## Current Top-Level Layout

```text
BiliSpider/
├── .github/              GitHub Actions and repository automation
├── bilispider/           Core Python package
├── data/                 Local runtime data, ignored and private
├── datasets/             Sanitized shared JSONL datasets
├── docs/                 Project documentation
├── examples/             Small runnable examples
├── packaging/            Windows packaging configuration and scripts
├── tools/                User-facing data and maintenance tools
├── tools/dev/            Developer-only reusable helper scripts
├── build/                Generated PyInstaller build output
├── dist/                 Generated PyInstaller distribution output
├── release/              Generated release installers and archives
├── gui.py                GUI entry point
├── login.py              QR login entry point
└── crawl_comments.py     CLI crawler entry point
```

## Directory Responsibilities

### `bilispider/`

Core application modules live here. Put reusable crawler, database, login, path, WBI, proxy, GUI package, and API logic in this package.

Do not put one-off scripts, raw API probes, local debug captures, or generated data here.

### Root Entry Points

The repository root should stay small. Stable launchers may live here:

- `gui.py`
- `login.py`
- `crawl_comments.py`

Do not add temporary `test_*.py`, `debug_*.py`, `analyze_*.py`, or `cdp_*.py` files to the root. Use `tmp/` or `tools/dev/` instead.

Developer benchmarks, API probes, and repair scripts are not stable root entry points. Keep reusable ones under `tools/dev/` with descriptive names such as `benchmark.py` or `probe_comment_api.py`.

### `tools/`

Put reusable command-line utilities here when they are meant for users or maintainers, for example:

- exporting/importing comments
- validating datasets
- producing dataset reports
- checking local database stats

Keep tools self-contained and avoid embedding machine-specific paths.

### `tools/dev/`

Put developer-only but reusable diagnostics here. Good examples:

- API compatibility probes
- manifest rebuild helpers
- controlled debug scripts that do not contain secrets
- local benchmarks used by maintainers

If the script is throwaway, use `tmp/` instead.

### `tmp/`

Use this for short-lived local experiments. It is ignored by git.

Examples:

- one-off API response checks
- temporary SQLite databases
- raw logs with local machine details
- throwaway scripts for a single investigation

Delete files here when the investigation is done.

### `docs/`

Put long-form documentation here:

- architecture notes
- release notes and changelogs
- data contribution instructions
- API behavior notes
- project structure rules

Use `docs/reference/` for external references or longer background material.

### `examples/`

Examples should be small and safe to run. They should show public usage patterns, not private experiments.

Do not place tests or debug probes here.

### `packaging/`

Windows packaging files live here:

- `BiliSpider.spec`
- `BiliSpider.iss`
- `build_windows.ps1`

Generated packaging output belongs in `build/`, `dist/`, or `release/`, never in `packaging/`.

### `datasets/`

Only sanitized collaboration data belongs here:

- `*.jsonl`
- `manifest.json`
- organized exports such as `datasets/by_uid/` and `datasets/by_oid/`

Do not put SQLite databases, cookies, raw captures, local scratch exports, or personal data here.

### `data/`

Runtime files live here during development:

- `cookies.json`
- `config.json`
- `crawl_queue.json`
- `comments.db`

These are local and private. Do not commit them.

Packaged builds use `%APPDATA%\BiliSpider\data` unless `BILISPIDER_DATA_DIR` overrides it.

### `build/`, `dist/`, `release/`

These are generated by packaging commands. Do not hand-edit files inside them. Rebuild instead.

## Naming Rules

- Stable tools: use descriptive verbs, for example `export_comments.py`.
- Developer diagnostics: use explicit names under `tools/dev/`, for example `probe_reply_cursor.py`.
- Temporary scripts: use `tmp/<purpose>.py`.
- Dataset files: use `comments_all_YYYY-MM-DD.jsonl`, `comments_uid_<uid>.jsonl`, or `comments_oid_<oid>.jsonl`.
- Avoid vague names like `test.py`, `new.py`, `fix.py`, `final.py`, or `script.py`.

## Security Rules

Never commit:

- Cookies, tokens, or authorization headers
- raw browser network captures with request headers
- `data/*.json` runtime configuration
- `data/*.db`, `*.db-wal`, `*.db-shm`
- files copied from a real browser profile

When documenting network research, record only:

- endpoint path
- non-sensitive query parameter names
- pagination structure
- response field names
- sanitized examples

## Adding A New Directory

Only add a new top-level directory when none of the existing directories fit. If you add one, update this guide and explain:

- what belongs there
- what does not belong there
- whether it is committed or generated
- whether it can contain sensitive data
